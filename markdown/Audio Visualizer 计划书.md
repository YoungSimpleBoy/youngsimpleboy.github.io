# Audio Visualizer 计划书

本文档用于规划 `toolkit/audio_visualizer/` 这套网页音频可视化的下一阶段演进方向。当前页面已经具备基础播放、频谱、波形、时频图、主题切换和鼓点脉冲等能力，但整体表现仍偏向“常规音频分析面板”。如果目标是做出更接近 TouchDesigner 的视觉冲击力，就不能只在二维图表上继续加细节，而要把音频特征、空间结构、颜色系统、时间延迟和反馈回路组织成一个实时视觉系统。

相关代码入口如下：

- [页面结构](../toolkit/audio_visualizer/index.html)
- [主逻辑](../toolkit/audio_visualizer/app.js)
- [样式文件](../toolkit/audio_visualizer/styles.css)

## 1. 项目目标

### 1.1 核心目标

把现有的网页音频可视化，从“播放音频时显示分析图表”升级为“实时音频驱动的视觉场景”。

最终要达到的效果不是单纯更花哨，而是让画面具备以下特征：

- 音频数据能驱动空间结构变化。
- 鼓点、瞬态、低频能形成明确的离散视觉事件。
- 连续频谱能驱动形变、色彩、光强、运动速度。
- 视觉不是一帧一帧的静态图，而是带有历史、延迟、残影和反馈。
- 页面在浏览器里仍能运行，但审美目标更接近 TouchDesigner 的实时装置感。

### 1.2 不做什么

为了避免目标发散，本阶段不优先做这些事：

- 不继续堆更多传统条形图控件。
- 不把页面做成纯数据仪表盘。
- 不追求“所有参数都暴露给用户”而牺牲视觉完整性。
- 不为了 3D 而 3D，所有空间元素都必须有音频映射依据。

## 2. 为什么现有方案不够

当前实现已经能展示频谱、波形和时频图，但它本质上还是“二维信息的多视图组合”。这类页面可读性不错，但很难给人“眼前一亮”的感觉，原因主要有三点：

1. 数据层次还不够立体。现在更多是单一音频流对应单一可视化层，缺少多个数据通道的耦合。
2. 时间维度还不够强。虽然有残影、脉冲和滚动图，但整体仍偏瞬时刷新，缺少延迟线、历史轨迹和反馈结构。
3. 空间语言还不够统一。波形、频谱、时频图、粒子等元素之间还没有形成一个统一的场景逻辑。

如果要接近 TouchDesigner 的观感，重点应该从“画图”转向“构建一个视觉合成管线”。

## 3. 设计原则

### 3.1 数据优先

每个视觉效果都必须能追溯到明确的数据来源，不允许纯装饰堆砌。推荐的数据源包括：

- RMS 能量
- 低频 / 中频 / 高频能量
- Spectral centroid
- Spectral flux
- Onset / beat confidence
- 峰值保持与衰减曲线
- 历史特征缓冲

### 3.2 连续量和离散量分工

建议把音频特征分成两类：

- 连续量：驱动形变、亮度、速度、颜色渐变、深度位移。
- 离散量：驱动脉冲、闪烁、爆发、粒子重排、镜头响应。

这样可以避免所有视觉都被同一套参数平均化，画面会更有节奏感。

### 3.3 先统一语言，再扩展内容

所有视觉层都要共享同一套色彩逻辑、节奏逻辑和空间逻辑。否则即使增加了很多效果，也会看起来像几个零散 demo 拼在一起。

### 3.4 保持网页可实施性

Web 端当然比 TouchDesigner 有限制，但限制不等于不能做。关键是要把实现路径分成可验证的小步，而不是一次性试图复制完整的 TD 生态。

## 4. 总体架构

建议把系统拆成五层：

### 4.1 音频数据层

负责从 Web Audio API 获取和整理音频特征。

建议保留并扩展：

- 时域数据
- 频域数据
- 低中高频分段能量
- 节拍触发判断
- 平滑后的历史值

建议新增：

- 音频特征历史环形缓冲
- 帧间增量特征
- 峰值衰减器
- beat 置信度
- 视觉驱动参数快照

### 4.2 映射层

负责把数据变成视觉参数，而不是直接画图。

映射关系示例：

- 低频 -> 体积膨胀、粒子重心下沉、镜头震动
- 中频 -> 点云密度、丝带曲率、轨迹弯折
- 高频 -> 亮点闪烁、边缘抖动、颜色尖峰
- beat -> 事件脉冲、闪光、后效反馈
- spectral flux -> 视觉转场强度、形状重组速度
- centroid -> 色相偏移与空间重心偏移

### 4.3 场景层

负责真正的视觉结构。

建议分为几个可叠加图层：

- 背景能量场
- 核心空间体
- 点云 / 粒子层
- 轨迹与反馈层
- 波形 / 频谱信息层
- 时频历史层
- 事件脉冲层

### 4.4 时间层

负责“让画面记住过去”。这是和普通网页可视化拉开差距的关键。

建议引入：

- 历史帧缓存
- 残影拖尾
- 延迟线
- beat 后效延迟
- 轨迹积分
- 速度缓动与衰减

### 4.5 UI 层

负责交互，但不抢视觉主导权。

UI 要做的事情只有三类：

- 选择音频与控制播放
- 切换预设与参数组
- 显示少量关键状态

不建议把参数面板做得过重，否则会把视觉系统拉回“设置页”的感觉。

## 5. 推荐的视觉方向

### 5.1 方向一：点云空间驱动

这是最接近 TouchDesigner 的方向之一。

思路是把音频特征映射到三维点云：

- 粒子位置由音频特征决定。
- 粒子速度和深度由低频和 beat 驱动。
- 颜色由频谱质心与频段能量驱动。
- 局部闪烁由瞬态和高频驱动。

如果最终技术栈允许，建议优先考虑 WebGL、Three.js 或 regl，而不是继续把 Canvas 2D 当主舞台。

### 5.2 方向二：反馈型能量场

这种方向更偏合成器视觉。

特点是：

- 画面里有不断循环的形状和残影。
- 音频能量改变反馈增益。
- 鼓点出现时，画面短暂变亮或重排。
- 形状随时间被拉扯、扭曲和重建。

这个方向对“高级感”很有效，而且比完整 3D 更容易先落地。

### 5.3 方向三：时频图升级为视觉叙事

现有时频图可以保留，但它不应只是辅助图表，而应成为“视觉历史带”。

可以把时频图改造成：

- 时间向右推进的能量地形图
- 颜色随频段和强度变化
- 与主场景共享同一套色彩系统
- 可作为背景纹理或下层信息带

这样时频图就不是“附属图表”，而是场景的一部分。

## 6. 技术路线建议

### 6.1 短期路线：在现有页面上增强

如果目标是尽快验证可行性，建议先在当前页面里做这几件事：

- 增加更完整的音频特征提取。
- 强化 beat 和瞬态的事件层。
- 引入历史轨迹、残影和延迟线。
- 把现有波形、频谱、时频图统一到同一套色彩和节奏系统。
- 试着加一个更强的“核心空间体”视觉中心。

这个阶段的价值在于：成本低，能快速判断哪些映射有效。

### 6.2 中期路线：引入 WebGL 场景

如果要明显提升冲击力，建议把核心视觉迁移到 WebGL。

可选技术：

- Three.js：适合快速搭建空间和材质。
- regl：适合写更干净的 GPU 渲染管线。
- 原生 WebGL2：适合最强控制力，但开发成本高。

推荐顺序：先用 Three.js 做原型，再决定是否需要更底层的方案。

### 6.3 长期路线：模块化视觉合成器

最终可以把项目做成一个轻量的网页视觉合成器：

- 音频输入模块
- 特征分析模块
- 视觉映射模块
- 场景渲染模块
- 预设管理模块
- 导出和分享模块

这样后续不只是能做一个页面，而是能持续做不同风格的音频视觉作品。

## 7. 里程碑计划

### 阶段 0：现状整理

目标：把现有代码状态整理清楚，避免在混乱结构上继续加功能。

任务：

- 梳理 `app.js` 里已有的数据流。
- 标出哪些参数是“分析参数”，哪些是“视觉参数”。
- 统一样式变量和布局命名。
- 明确当前页面里哪些模块必须保留，哪些可以重做。

完成标志：代码结构可以被清楚分层，后续功能不会互相打架。

### 阶段 1：高价值特征增强

目标：让音频驱动更像“有生命的系统”。

任务：

- 添加更多频段统计。
- 计算 spectral centroid、flux、peak decay 等特征。
- 优化 beat 触发逻辑，减少误判和迟滞。
- 建立历史缓冲，支持视觉延迟和惯性。

完成标志：画面对节奏和音色变化的响应更丰富，而不是只盯着平均能量。

### 阶段 2：空间视觉原型

目标：做出第一个真正有“空间感”的原型。

任务：

- 设计点云或粒子空间。
- 建立音频特征到坐标的映射。
- 设计颜色、深度、速度、密度的联动规则。
- 加入镜头旋转、轻微透视或伪 3D 效果。

完成标志：画面不再是扁平图表，而是有前后层次和运动体积。

### 阶段 3：反馈与时间结构

目标：补上 TouchDesigner 最关键的“时间味”。

任务：

- 加入残影与拖尾。
- 做延迟线和回授层。
- 让鼓点触发瞬时变形和亮度爆发。
- 让历史状态影响当前帧，而不是每帧都重新开始。

完成标志：画面能明显“记住”刚刚发生过的声音。

### 阶段 4：场景整合

目标：把所有视觉模块统一成一个完整作品。

任务：

- 统一主题与预设。
- 统一 UI 样式。
- 统一移动端和桌面端的布局策略。
- 统一参数保存与分享方式。

完成标志：页面看起来像一个完整作品，而不是几个功能块拼接起来的工具。

## 8. 具体功能清单

### 必做

- 音频特征增强
- 鼓点与瞬态事件层
- 视觉历史与残影
- 统一的配色系统
- 更具空间感的主视觉
- 性能保护和降级策略

### 可选但很有价值

- 预设系统
- 参数保存到 URL 或 localStorage
- 分享链接
- 全屏演示模式
- 导出静态帧或动图方案

### 不建议优先做

- 继续堆大量传统统计图
- 过多独立面板
- 一次性做复杂编辑器
- 为少量参数增加过重 UI

## 9. 风险与应对

### 风险 1：视觉太散

如果同时上太多层，会变成“很热闹但没有焦点”。

应对：

- 所有视觉都围绕同一个核心体展开。
- 限制主色系数量。
- 只让少数几个音频特征拥有最高权重。

### 风险 2：性能下降

WebGL 或高密度粒子容易带来性能压力。

应对：

- 设置帧率保护。
- 低性能设备自动降粒子密度。
- 分离分析频率和渲染频率。

### 风险 3：交互压过视觉

如果参数太多，用户会回到“调表”的状态。

应对：

- 保留少数关键控制项。
- 把大部分复杂性收进预设。
- 让默认效果足够好看。

### 风险 4：网页限制导致落差

Web 端确实不如 TouchDesigner 那样直接。

应对：

- 接受 Web 的优势是可分享、可部署、可交互。
- 用模块化设计换取灵活度。
- 优先做“视觉语言接近”，不执着于完全复刻工作流。

## 10. 建议实施顺序

如果后面真的开始做，建议按这个顺序推进：

1. 整理现有代码与布局。
2. 扩展音频特征层。
3. 做一个空间感更强的主视觉原型。
4. 加反馈、延迟和轨迹。
5. 再决定是否迁移到 WebGL 核心渲染。
6. 最后才做大范围样式统一和分享功能。

这个顺序的好处是：先验证“有没有惊艳感”，再决定是否投入更重的技术栈。

## 11. 结论

这套网页音频可视化如果继续沿着现有二维图表方向发展，很难获得真正强烈的视觉记忆点。更合理的方向，是把它升级成一个“音频驱动的实时视觉系统”：让音频特征成为场景生成逻辑，让时间延迟、反馈回路和空间结构共同参与画面生成。

换句话说，这个项目的下一个阶段不应该只是“更好看的音频图”，而应该是“网页里的 TouchDesigner 风格合成器”。

如果后续要实施，建议先从现有页面中抽离出音频特征与视觉映射的边界，再决定是继续增强 Canvas 2D，还是切到 WebGL 作为主渲染层。
